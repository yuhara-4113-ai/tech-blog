## はじめに

子どもとスマホアプリの「ポケポケ」にハマったことをきっかけに、家族の写真でポケモンカードをオマージュしたオリジナルカードを作れるWebアプリを開発しました。

完成したのが[Original Card Studio](https://yuhara-4113-ai.github.io/original-poke-card-generator/)です。名前、属性、技、説明文、画像などを設定すると、入力内容がリアルタイムでカードに反映され、完成したカードを高解像度PNGで保存できます。

![Original Card Studioの画面](/images/build-card-generator-with-codex-from-smartphone/app-overview.png)

- 公開URL: [Original Card Studio](https://yuhara-4113-ai.github.io/original-poke-card-generator/)
- GitHub: [yuhara-4113-ai/original-poke-card-generator](https://github.com/yuhara-4113-ai/original-poke-card-generator)
- 使用技術: React 19、Vite 8、JavaScript、SVG、Canvas API、CSS、GitHub Actions、GitHub Pages

今回の特徴は、実装の大部分をCodexに任せ、私は主にスマートフォンから自然言語で指示したことです。寝ながらでも、出先でも、スマホから開発を進められました。

この記事では、単に「AIに作ってもらった」で終わらせず、どのように要件を伝え、公開可能なWebアプリへ仕上げたのかを紹介します。

## 1年前はGitHub Copilotで形にできなかった

実は、約1年前にもGitHub Copilotを使って同じアイデアに挑戦していました。しかし、当時の私の使い方では、期待した見た目や操作感まで仕上げられず、実用できる状態にはできませんでした。リポジトリに残っていたのも、READMEとライセンスだけでした。

今回はCodexに「この機能のコードを書いて」と断片的に依頼するのではなく、リポジトリ全体を対象に、調査、実装、検証、修正、Git操作まで一連の作業を依頼しました。その結果、最初の動くバージョンは短時間で形になり、そこから数日間の改善で公開・運用できる状態まで進められました。

もちろん、これはCopilotとCodexの一般的な性能を比較したベンチマークではありません。1年前と現在では、AIの機能だけでなく、私自身の依頼方法も変わっています。それでも、以前は形にできなかったアイデアが、今回は驚くほど早く動き始めたことに大きな変化を感じました。

## どのようにスマホから開発したのか

開発中、私はコードをほとんど直接編集していません。代わりに、次のような内容を日本語でCodexへ伝えました。

- 作りたい機能と、その背景
- ユーザーにどのように操作してほしいか
- 実際に使って感じた違和感
- 修正後に確認してほしい画面幅や操作
- ブランチ作成、コミット、Push、Pull Request作成までの作業範囲

例えば、画像位置の調整は当初スライダー操作でした。しかし、スマホで使うと直感的ではなかったため、次のように依頼しました。

> 画像の位置調整を直感的にしたい。今はスライダーだけど、画像の位置を直接調整できるようにしたい。

Codexは既存の実装を確認し、プレビュー上の画像をドラッグして位置を変えられるように修正しました。内部では、すでに存在していた画像位置のstateを再利用しているため、描画処理全体を作り直してはいません。

また、スマホで本番反映前に確認したかったため、次のようにも依頼しました。

> 本番以外に検証環境を作ってほしい。スマホで検証環境にアクセスし、問題がなければマージして本番環境にデプロイしたい。

これにより、`codex/**`ブランチへPushすると、GitHub Pagesの`/staging/`へ検証版がデプロイされる構成になりました。私はスマホで実際のURLを開き、表示や操作を確認してからPull Requestをマージできます。

```text
スマホから自然言語で指示
        ↓
Codexがリポジトリを調査・実装
        ↓
lint / production build / ブラウザ確認
        ↓
codex/** ブランチへPush
        ↓
GitHub Pagesのstagingをスマホで確認
        ↓
Pull Requestをマージして本番公開
```

## アプリの構成

このアプリは、バックエンドを持たないReact製のSPAです。入力内容とアップロード画像はブラウザ内で扱い、カード本体をSVGで描画しています。

```text
PokemonCardGenerator（カード情報をstateで管理）
├── CardForm（入力）
├── PokemonCard（プレビュー、傾き、光沢）
│   └── CardArtwork（SVGでカードを描画）
└── DownloadButton（SVGをCanvas経由でPNG化）
```

フォームから受け取った値を親コンポーネントのstateへ集約し、同じデータをSVGのプレビューとPNG出力の両方で使います。プレビュー用と保存用に別々のカードを実装していないため、文字や画像の位置がずれにくい構成です。

### SVGとCanvas APIによるPNG出力

カードは`660 × 921`の固定座標を持つSVGとして描画しています。保存時はプレビュー中のSVGを複製し、属性アイコンなどの外部参照画像をData URLとして埋め込んだ後、Canvasへ描画します。

Canvasを2倍の`1320 × 1842 px`にすることで、高解像度PNGを生成しています。光沢ありで保存する場合は、Canvas上で光沢表現も合成します。

カードのインタラクティブな光沢表現は、[@simeydotme/pokemon-cards-css](https://github.com/simeydotme/pokemon-cards-css)を参考にしています。ライセンスを確認し、このアプリもGPL-3.0で公開しています。

### 画像はブラウザ内だけで処理

アップロード画像は`FileReader`でData URLへ変換し、Reactのstateとして保持します。バックエンドへ送信する処理はありません。

## GitHub Actionsで検証と公開を自動化

`main`ブランチへ反映すると、GitHub Actionsが次の処理を実行します。

1. `npm ci`
2. `npm run lint`
3. `npm run build`
4. 成功した成果物をGitHub Pagesへデプロイ

検証環境用のワークフローでは、`main`から本番版、作業ブランチから検証版をそれぞれビルドし、1つのGitHub Pages成果物へまとめています。

```text
site/
├── 本番版
└── staging/
    └── 検証版
```

本番URLを壊さず、スマホから実機に近い状態で確認できるため、PCを開けない時間でもレビューを進められました。

## AI主導開発で人が担当したこと

コードの入力をCodexに任せても、開発者の仕事がなくなるわけではありませんでした。今回、私が特に意識したのは次の点です。

### 1. 「何を作るか」ではなく「どう使えてほしいか」を伝える

「画像位置を変更する機能を追加して」だけでは、スライダー、数値入力、ドラッグ操作など複数の実装が考えられます。

そこで、「スマホで画像を直接動かしたい」「本番反映前に実際のURLで確認したい」のように、期待するユーザー体験まで伝えました。実装方法を細かく指定するよりも、目的と制約を具体的に共有した方が、Codexから適切な提案を得やすいと感じました。

### 2. 生成結果を実際に触って修正する

最初の出力がそのまま完成版になったわけではありません。スマホで操作し、入力欄の重なり、文字の位置、ボタン名、光沢の初期値など、気になった点を追加で依頼しました。

特にUIは、コード上で正しく見えても実際の画面では違和感が出ます。Codexにも`390 × 844`相当の画面でブラウザ確認を依頼し、lintやbuildだけでは見つからない問題を修正しました。

### 3. AIの変更範囲を決める

例えば入力内容の保存では、「画像以外でよい」「一旦バージョン互換は考えなくてよい」と範囲を絞りました。要件を広げすぎないことで、必要な機能を小さく実装し、確認しやすくなりました。

## うまくいかなかった点と注意点

スマホから指示できるのは便利ですが、すべてを一度に依頼すると確認が難しくなります。実際には、次の単位で小さく進める方が安定しました。

1. 1つの違和感や機能を伝える
2. Codexに既存実装を調査してもらう
3. 実装後にlint、build、ブラウザで確認する
4. stagingを自分のスマホで触る
5. 問題がなければマージする

また、AIが生成したコードだからといって、そのまま正しいとは限りません。固定座標を多用するSVGでは、一箇所の変更が別のレイアウトに影響することがあります。通常版と全面アート版、日本語と英語、PCとスマホなど、影響範囲を言葉にして確認させることが重要でした。

## まとめ

約1年前、GitHub Copilotを使った挑戦では形にできなかったアプリを、今回はCodexを実装担当として活用し、公開できるところまで仕上げられました。

最も驚いたのは、PCの前に座ってコードを書き続けなくても、寝ながらでも出先でも、スマホから要件や修正内容を伝えて開発を進められたことです。アイデアを思いついた瞬間に指示し、検証URLをスマホで触り、次の修正を返せます。

一方で、AI主導開発でも重要なのは、目的、制約、期待する体験を具体的に言語化し、生成物を確認して判断することでした。コードを書く時間は大きく減りましたが、何を良しとするかを決める役割は人に残ります。

AIの進化によって、個人のアイデアを実際に触れるプロダクトへ変えるハードルは確実に下がっていると感じました。今後、さらにどこまでスマホだけで開発できるようになるのか楽しみです。
